# 设计决策（ADRs）

记录项目中的关键选型决策，避免以后忘了"当时为什么这么选"。

---

## ADR-001: 用 MySQL 而不是 SQLite

**时间：** 2026-06-28

**背景：**
原项目用 JSON 文件当数据库，推完即焚，没有历史记录。需要引入真正的数据库。

**选项：**
- SQLite：零配置、单文件、Python 自带
- MySQL：需要装服务但阿杰熟悉
- PostgreSQL：理论最优但需要学习

**决策：** MySQL

**理由：**
- 阿杰熟悉 MySQL，出问题能自己修
- 未来有商业扩展空间，MySQL 在百万级数据量下依然稳定
- 不做性能最优选型，做开发者最舒服的选型

**后果：**
- 服务器需要安装 MySQL 服务（ECS 内存可能紧张）
- 数据量大时考虑切 PostgreSQL 或加 ES

## ADR-002: 不用 ORM，直接原生 SQL

**背景：**
需要操作 MySQL，但 SQLAlchemy 的 async 支持比较复杂。

**选项：**
- SQLAlchemy + aiomysql：成熟但重量
- 直接 aiomysql + 原生 SQL：简单直接
- Tortoise ORM：轻量 async ORM

**决策：** 直接 aiomysql + 原生 SQL

**理由：**
- 阿杰说 SQL 熟，那就用 SQL
- 6 张表、查询不复杂，ORM 带来的抽象收益有限
- 减少一层依赖，排查问题更直接

## ADR-003: 新建 StockGod 项目，不改原项目

**背景：**
原 `stock` 项目正在线上运行（noohup 跑在 ECS），不能中断。

**选项：**
- 在原项目上改：容易搞崩线上服务
- 新建项目：干净，但代码要重写

**决策：** 新建 StockGod 项目

**理由：**
- 原项目稳定运行，不要动
- 新建项目可以从零设计，不受遗留代码束缚
- 后期切换方式：新项目部署后停掉旧进程，切流量

## ADR-004: ChromaDB 做向量库

**背景：**
RAG 需要向量数据库存储 LLM 分析结果的 embedding。

**选项：**
- ChromaDB：零配置、文件级、Python 原生
- Milvus：分布式、强、但重
- Pinecone：托管服务、要付费
- Elasticsearch：也可以做向量搜索，但重

**决策：** ChromaDB

**理由：**
- 数据量小（几十万条以内），ChromaDB 足够
- 零配置，不需要额外维护一个服务
- 够用就行，等数据量到了百万级再换

## ADR-005: 不做 K 线图等看盘功能

**背景：**
前端设计时考虑过做 K 线图、实时走势等看盘功能。

**方案：** 不做

**理由：**
- 看盘用东方财富/同花顺，不可能做得比它们好
- 自己系统只做"复盘和统计"——异动看板、信号准确率、涨停分析
- 做跟它们不一样的事，不是在它们的赛道上竞争
